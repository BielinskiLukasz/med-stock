## Interaction & Animation

### Search (Medicines Screen, Top)

1. **Default state:** Search bar visible, empty, placeholder "Search medicines by name…"
2. **User types:** Results filter instantly (no button submit), substring match case-insensitive on `medicine.name`
3. **Clear:** User can clear search via X icon on right side of input OR manual deletion of text
4. **Combined with filters:** Search AND filter results apply simultaneously (AND logic)

### Filter/Sort Controls (Bottom Sheet)

1. **Trigger:** Filter icon in top-right of Medicines screen header
2. **Behavior:** Slide-up bottom sheet with filter/sort options
3. **Filter dimensions:** Category (multi-select, OR logic within), Location (multi-select, OR logic within), Status (multi-select, OR logic within)
4. **Sort dimensions:** Name (A-Z), Expiry Date (soonest first), Category (A-Z)
5. **Apply:** Changes apply instantly when user selects/deselects options (no "Apply" button)
6. **Dismiss:** Tap outside, swipe down, or press Escape
7. **State persistence:** Filter state persists in Zustand store across navigation

### Filter Chips (Above Medicine List)

1. **Display:** Horizontal row of chips, each showing one active filter
2. **Format:** "{Dimension}: {Value} ×" (e.g., "Location: Bathroom ×")
3. **Dismiss:** Tap × on any chip to remove that filter
4. **Count badge:** Filter icon shows badge with active filter count (e.g., "2 active")

### Dashboard

1. **Layout:** 2x2 grid of 4 cards (Total, Expired, Expiring Soon, Exceeded Open Period)
2. **Card style:** White background, shadow, rounded corners, 16px padding
3. **Total card:** Always present, non-interactive (summary only)
4. **Alert cards:** Tappable; tap opens Medicines screen with that filter applied
5. **Visual hierarchy:** Expired card in red (destructive color), Expiring Soon in amber, Exceeded in orange

### Trash Bin

1. **Empty state:** "Trash is empty" centered text if no deleted medicines
2. **Deleted medicines list:** MedicineCard-style layout (same as search results)
3. **Buttons per item:** "Restore" button (primary blue) and "Delete Permanently" button (destructive red)
4. **Restore action:** Removes `deletedAt` timestamp, restores medicine to main list with previous status
5. **Delete Permanently action:** Hard delete after confirmation; history is preserved

### Change History (Detail View)

1. **Location:** Section at bottom of medicine detail view
2. **Trigger:** Collapsible section header "Change History" (user can expand/collapse)
3. **Timeline:** Vertical list of entries, each with timestamp and action summary
4. **Entry format:** "{timestamp} — {action}" (e.g., "2026-06-30 10:15 — Name changed from 'Ibuprofen' to 'Ibuprofen 400mg'")
5. **Preserved after deletion:** History entries remain visible even if the medicine is permanently deleted

## Responsive Design

**Mobile-first (primary target):**
- Medicines screen: Full viewport width, search bar sticky at top
- Dashboard: 2x2 grid on phones (4 cards stacked vertically or in pairs)
- Trash: Full-width list, same as medicines
- Detail view: Full viewport, scrollable, history section at bottom

**Tablet+ (future — not Phase 2 scope):**
- Layout may widen to use more space, but no changes to interaction model are required for Phase 2.

## Implementation Notes

**For Executor:**

1. **SearchBar component:** Use shadcn `Input` with `type="text"`, add clear button (X icon) on right via onReset handler.
2. **FilterBottomSheet:** Use Radix Dialog with custom slide-up animation; state in Zustand `uiStore.ts`.
3. **Dashboard cards:** Tappable (use Link or onClick handlers); tap applies the filter to Medicines screen and navigates to it.
4. **Trash restore:** Call Dexie update to set `deletedAt = null`, then re-fetch with `useLiveQuery`.
5. **History table:** Dexie v2 schema already planned in 02-CONTEXT.md D-41; historian entries are immutable (insert-only).
6. **Status filtering:** Use `calculateStatus()` client-side after Dexie query returns (auto-calculated statuses must be post-filtered per D-12).

**For Planner:**

- **Color decisions:** No phase-specific colors needed; Phase 1 palette is sufficient (green=Active, blue=Opened, red=Expired, etc.).
- **Typography decisions:** No new font sizes or weights; reuse Phase 1 body (14px/400) and heading (16px/500).
- **Spacing decisions:** Standard 8-point scale; no exceptions required for Phase 2.
